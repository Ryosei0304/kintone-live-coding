# 2-Pass デプロイルール

## なぜ2パスが必要か

kintoneでは参照先アプリがデプロイ済みでないとルックアップ/関連レコードを設定できない。
このプロジェクトでは **2-Pass デプロイ** でこの問題を解決する。

## Pass 1: 基本フィールド

1. 全アプリを作成（プレビュー状態）
2. 基本フィールド（ルックアップ/関連レコード以外）を追加
3. 全アプリをデプロイ

## Pass 2: 連携フィールド（優先度順）

1. ルックアップフィールドを優先度順に追加
2. 関連レコード一覧フィールドを追加
3. フォームレイアウトを最適化
4. 全アプリを再デプロイ

## デプロイ優先度

以下の優先度に従ってデプロイ順序を決定する。

1. **マスタアプリ優先**: 他のアプリから参照されるマスタアプリを先にデプロイ
2. **参照先 → 参照元の順**: ルックアップ/関連レコードの参照先アプリが先、参照元アプリが後
3. **循環参照はエラー**: 循環参照がある場合はエラーとする

## 進捗表示例

```
[Pass 1] 基本フィールド
  アプリ作成 (1/3): 顧客マスタ
  アプリ作成 (2/3): 商品マスタ
  アプリ作成 (3/3): 受注管理
  デプロイ: 完了

[Pass 2] 連携フィールド（優先度順）
  顧客マスタ → 企業マスタ (ルックアップ): OK
  受注管理 → 顧客マスタ (ルックアップ): OK
  受注管理 → 商品マスタ (ルックアップ): OK
  デプロイ: 完了
```

## 5-Pass アップデートデプロイ（`/restart` 用）

既存アプリへの変更適用時は 5-Pass デプロイを使用する。

### Pass U1: 基本フィールド変更

1. 既存アプリの基本フィールドを変更（ルックアップ/関連レコード以外）
   - DELETE → MODIFY → ADD の順（依存関係を最小化）
2. 全対象アプリをデプロイ

### Pass U2: 新規アプリ + 関係変更（依存順）

1. 新規アプリがある場合:
   - アプリ作成 + 基本フィールド追加 → デプロイ
2. ルックアップ/関連レコードの追加・変更:
   - マスタアプリ優先、参照先 → 参照元の順
   - 各ルックアップ追加後に中間デプロイ
3. 中間デプロイ

### Pass U3: プロセス管理更新

1. プロセス管理の追加・変更
2. 全対象アプリをデプロイ

### Pass U4: レイアウト最適化

1. フォームレイアウト更新（横並び最適化、スペーサー・ラベル配置）
2. 全対象アプリをデプロイ

### Pass U5: ビュー更新

1. ビュー追加・変更
2. 最終デプロイ

### カスタマイズ保持ルール

- 変更計画で言及されていないカスタマイズは既存の `fileKey` をそのまま保持
- 新規カスタマイズのみアップロード・追加
- 既存JSファイルの上書きは禁止

### 進捗表示例

```
[Pass U1] 基本フィールド変更
  顧客マスタ (DELETE): phone_old ... OK
  顧客マスタ (ADD): email, phone_new ... OK
  受注管理 (MODIFY): status ... OK
  デプロイ: 完了

[Pass U2] 新規アプリ + 関係変更
  新規アプリ作成: 活動履歴 (ID: 456) ... OK
  デプロイ: 完了
  ルックアップ追加: 活動履歴 → 顧客マスタ ... OK
  デプロイ: 完了

[Pass U3] プロセス管理更新
  受注管理: プロセス管理更新 ... OK
  デプロイ: 完了

[Pass U4] レイアウト最適化
  顧客マスタ: レイアウト更新 ... OK
  受注管理: レイアウト更新 ... OK
  活動履歴: レイアウト設定 ... OK
  デプロイ: 完了

[Pass U5] ビュー更新
  受注管理: ビュー追加 (進捗ビュー) ... OK
  活動履歴: ビュー追加 (最新活動) ... OK
  最終デプロイ: 完了
```
