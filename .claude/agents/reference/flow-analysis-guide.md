# 業務フロー分析ガイド

kintone-flow-analyst エージェントが使用する業務フロー分析の詳細手順。

## 原則

**重要**: 「聞く」より「見せて直す」方が速い。
コンテキスト整理の情報からAs-Isフローを**推測して提案**し、ユーザーに修正してもらう。

---

## Step 1: コンテキスト読み込み

`コンテキスト整理_${Project}_${Date}.md` を読み込み、以下を把握:
- アクター一覧
- ペインポイント
- ゴール
- スコープ（仮）

---

## Step 2: As-Is 業務フロー推測 → 提案

コンテキストから現状の業務フローを推測し、Mermaid sequenceDiagram で提案する。

**提案フォーマット**:
```
お話を聞いた内容から、今の業務の流れをこんな感じで理解しました！

[Mermaid図]

合っていますか？違うところがあれば教えてください！
```

**推測のポイント**:
- アクター間のやり取り（誰が誰に何を渡すか）
- 使っているツール（Excel、メール、紙など）
- 課題が発生するポイントを `Note` で注記

→ ユーザーの修正を反映してAs-Isフローを確定

---

## Step 3: To-Be フロー提案

kintone導入後のTo-Beフローを提案する。

**As-Is → To-Be の変換ルール**:
| As-Isパターン | To-Beパターン |
|-------------|-------------|
| Excel管理 → メール送信 | kintoneアプリで一元管理 + 通知 |
| 紙の申請 → 手渡し承認 | kintoneプロセス管理で電子承認 |
| 口頭依頼 → 個人メモ | kintoneレコード作成 + 担当者割当 |
| 集計作業（手動） | kintoneグラフ・集計機能 |

**提案フォーマット**:
```
kintoneを使うと、こんな流れになります！

[Mermaid図]

改善ポイント:
1. ${改善点1}
2. ${改善点2}

この流れでよさそうですか？
```

→ ユーザーの修正を反映してTo-Beフローを確定

---

## Step 4: ビジネスイベント洗い出し

To-Beフローから「データが生まれるトリガー」を洗い出す。

**ビジネスイベントの見つけ方**:
- フロー図の中で「レコード作成」が発生するポイント
- ステータスが変わるポイント
- 外部からデータが入ってくるポイント

**提示フォーマット**:
```
この業務で「データが生まれるタイミング」を整理しました。
これらがアプリの元になります！

| No | イベント名 | トリガー | 生成されるデータ |
|----|-----------|---------|----------------|
| 1  | ... | ... | ... |

漏れがないか確認してもらえますか？
```

---

## Step 5: MVPスコープ線引き

全機能をMVP（初回リリース）と将来対応に分類する。

**MVP判定基準**:
| 判定 | 条件 |
|------|------|
| MVP | ペインポイントの直接解決に必要 |
| MVP | 業務が回るための最低限の機能 |
| 将来 | あると便利だが、なくても業務は回る |
| 将来 | 複雑な連携や高度な機能が必要 |

---

## Step 6: 規模感・連携・セキュリティ確認

業務フローが固まった段階で、ここで初めて聞く。

**質問テンプレート**:
```
業務フローが整理できたので、あと少しだけ確認させてください！

1. 利用者数はだいたい何人くらいですか？
2. データは月にどのくらい増えそうですか？
3. 他のシステムとの連携は必要ですか？
4. アクセス制限で気をつけることはありますか？

分からないものは「分からない」で大丈夫です！
```

---

## Step 7: カスタマイズ提案

業務フローが固まった段階で、業務特性から適用可能なカスタマイズを推奨度付きで提案する。

### 推奨度の3段階

| 推奨度 | 意味 | 判定基準 |
|--------|------|----------|
| ★★★ 強く推奨 | 運用上ほぼ必須 | ステータス管理あり→完了時の編集ロック等 |
| ★★ 推奨 | この業務特性に合っている | 入力項目が多い→セクション見出し等 |
| ★ あると便利 | なくても回るが、あると嬉しい | 一覧の色分け、文字数カウント等 |

### 提案ロジック（業務特性から判定）

| 業務特性のヒント | 提案するカスタマイズ | 推奨度 |
|-----------------|-------------------|--------|
| ステータス管理・承認フローがある | 完了時にフィールド編集不可 (`field_disable`) | ★★★ |
| ステータスに応じて入力項目が変わる | ステータスで表示/非表示を切替 (`field_show_hide`) | ★★★ |
| 条件によって必須項目が変わる | 条件付き必須チェック (`validate_required`) | ★★★ |
| 管理項目が多い（10項目以上）業務 | セクション見出しで区切り (`style_section_header`) | ★★ |
| 電話番号・メール・郵便番号を扱う | 入力フォーマットチェック (`validate_format`) | ★★ |
| 明細行（見積明細、注文明細等）がある | テーブル合計計算 (`table_total`) | ★★ |
| 採番ルールがある（例: INV-001） | 自動採番 (`auto_numbering`) | ★★ |
| 入社日・契約開始日など期間管理 | 経過年数/日数表示 (`elapsed_years`) | ★ |
| 一覧画面でステータスを素早く把握したい | 一覧画面の条件書式 (`list_conditional_style`) | ★ |
| 備考欄など自由入力が多い | 文字数カウント表示 (`field_char_count`) | ★ |

### 提案フォーマット

```
業務の流れを整理した内容から、こんなカスタマイズが効果的だと思います！

| # | 推奨度 | カスタマイズ | 内容 | 理由 |
|---|--------|------------|------|------|
| 1 | ★★★ | フィールド編集不可 | ステータスが完了のとき全フィールドを編集不可に | 承認済みデータの誤編集を防止できます |
| 2 | ★★ | セクション見出し | 「基本情報」「連絡先」で区切り | 管理項目が多いので、セクションで区切ると見やすくなります |

この中で「いいね！」と思うものはありますか？
他にも「こういうことできる？」があれば気軽に聞いてください！
```

### 回答の処理

- 選択されたカスタマイズ → 業務フロー設計書のカスタマイズ要件セクションに記載
- 追加要望 → パターンカタログ（`templates/customize/catalog.json`）と照合し、対応可否を回答
- 全て不要 → カスタマイズ要件なしとして記録

---

## 出力の品質チェック

生成前に以下を確認:
- [ ] As-Is/To-Beの両方がフロー図付きで記述されている
- [ ] ビジネスイベントが漏れなく洗い出されている
- [ ] MVPスコープが明確に線引きされている
- [ ] カスタマイズ提案が推奨度付きで記載されている
