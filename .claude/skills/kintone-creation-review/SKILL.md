# Phase 3: デプロイエキスパートレビュー

デプロイ実行前に、設計書とkintone環境をエキスパートの観点でレビューします。

## 概要

このスキルはPhase 3（アプリ作成）の実行前、HITL（ユーザー確認）前に自動実行され、以下を検証します：

- API制限値の遵守
- デプロイ順序の妥当性
- プレビュー/本番環境の理解
- エラーハンドリングの準備
- デプロイ優先度の確認

## 入力

- Phase 2で生成された設計書ファイル
  - `アプリ設計書_${Project}_${Date}.md`
  - `フィールド設計書_${Project}_${Date}.md`
- kintone環境情報（MCP経由で取得）

## レビュー観点

### 1. API制限値の確認

| 制限項目 | 上限値 | チェック内容 |
|---------|-------|-------------|
| 1アプリへのリクエスト数 | 10,000リクエスト/日 | 大量データ登録時に超過しないか |
| ドメインごとの同時アクセス数 | 100 | 並列処理の設計は適切か |
| APIリクエストボディサイズ | 50MB | 添付ファイルのサイズは適切か |
| 1アプリのAPIトークン数 | 20個 | トークン管理は適切か |
| 一度に取得できるレコード数 | 500件 | ページング処理は実装されているか |
| 一度に登録/更新/削除できるレコード数 | 100件 | バッチ処理の設計は適切か |
| offsetの上限 | 10,000件 | 大量データ取得時の制限を認識しているか |

### 2. デプロイAPI仕様の確認

| API | エンドポイント | 説明 |
|-----|---------------|------|
| アプリ作成 | `POST /k/v1/preview/app.json` | プレビュー環境にアプリ作成 |
| フィールド追加 | `POST /k/v1/preview/app/form/fields.json` | プレビュー環境にフィールド追加 |
| レイアウト更新 | `PUT /k/v1/preview/app/form/layout.json` | プレビュー環境のレイアウト更新 |
| 本番反映 | `POST /k/v1/app/deploy.json` | プレビュー→本番への反映 |
| デプロイステータス確認 | `GET /k/v1/app/deploy.json` | 反映状況を確認（非同期処理） |

### 3. プレビュー環境と本番環境

| 環境 | URLパス | 説明 |
|------|---------|------|
| プレビュー（テスト） | `/k/v1/preview/xxx.json` | 設定変更を保存（未反映状態） |
| 本番（運用） | `/k/v1/xxx.json` | 実際に利用される環境 |

**重要**:
- プレビュー環境での変更は、デプロイAPIを実行するまで本番に反映されない
- デプロイは非同期処理のため、ステータス確認が必要

### 4. デプロイ優先度の確認

kintone-app-creationスキルでは、フィールド間の依存関係を優先度順に管理します。

#### 2パス作成方式

**Pass 1: 基本フィールド**
- ルックアップ/関連レコードを含まないフィールド
- アプリごとにまとめて作成可能
- デプロイ後、Pass 2に進む

**Pass 2: 連携フィールド**
- ルックアップ/関連レコードフィールド
- トポロジカルソート順に作成
- 参照先フィールドがデプロイ済みであることが前提

#### チェック項目

- [ ] 循環参照がないか（依存グラフが非循環か）
- [ ] 参照先フィールドがPass 1に含まれているか
- [ ] デプロイ順序が優先度に従っているか

### 5. デプロイ前チェックリスト

#### アプリ作成順序

- [ ] マスタアプリが先にデプロイ対象になっているか
- [ ] トランザクションアプリは参照先アプリのデプロイ後に作成されるか
- [ ] 循環参照がないか

#### フィールド設計

- [ ] フィールドコードが命名規則に従っているか
- [ ] フィールドコードに使用禁止文字が含まれていないか
- [ ] 必須フィールドにデフォルト値が設定されているか（必要な場合）
- [ ] ルックアップのキーフィールドが重複禁止設定か
- [ ] テーブル内フィールド数が適切か

#### アプリ設定

- [ ] アプリ名が重複していないか
- [ ] アプリの説明が設定されているか

### 6. デプロイ順序の最終確認

**正しいデプロイフロー（2パス方式）**:

```
Phase A: 全アプリの基本フィールド（Pass 1）
1. 全アプリを作成（プレビュー）
2. 基本フィールドを追加（プレビュー）
3. 基本フィールドをデプロイ（本番反映）★
4. デプロイ完了を確認

Phase B: 連携フィールド（Pass 2）
5. ルックアップ/関連レコードフィールドを優先度順に追加
6. レイアウトを更新
7. 連携フィールドをデプロイ（本番反映）★
8. デプロイ完了を確認
```

### 7. エラーハンドリング準備

| HTTPステータス | 意味 | 対応方法 |
|---------------|------|---------|
| 400 | リクエスト不正 | パラメータを確認、エラーメッセージを解析 |
| 403 | 権限エラー | APIトークン権限、IPアドレス制限を確認 |
| 404 | リソースが存在しない | アプリID、フィールドコードを確認 |
| 429 | 同時アクセス上限超過 | リトライ処理（指数バックオフ推奨） |
| 500 | サーバーエラー | 時間をおいて再実行 |
| 520 | アプリ作成上限 | 不要アプリの削除、契約プラン確認 |

### 8. トランザクション処理の注意点

**kintoneの特性**:
- 複数アプリへのAPI処理をまとめて1つのトランザクションにできない
- アプリAの登録成功後、アプリBの登録が失敗してもAはロールバックされない

**対策**:
- レコードのリビジョンを利用して同時編集を防止
- 重複禁止フィールドで二重登録を防止
- 仮ステータス → 本登録ステータスの2段階方式を検討

### 9. デプロイ後の検証項目

- [ ] アプリが正常に表示されるか
- [ ] レコードの登録・編集・削除が正常に動作するか
- [ ] ルックアップが正常に機能するか（取得ボタンで値が取れるか）
- [ ] 関連レコード一覧が正常に表示されるか
- [ ] プロセス管理（ワークフロー）が正常に動作するか
- [ ] 権限設定が正しく適用されているか
- [ ] フォームレイアウトが適切か

## レビュー実行手順

### Step 1: 設計書の読み込み

outputs/ディレクトリから設計書ファイルを読み込む。

### Step 2: kintone環境の確認

MCP経由で以下を確認：
- 既存アプリ一覧（名前の重複確認）
- 接続テスト

### Step 3: デプロイ優先度の検証

フィールド設計書から依存関係を確認し、以下を検証：
- 循環参照の有無
- Pass 1/Pass 2の分離が適切か
- デプロイ順序が妥当か

### Step 4: チェック項目の検証

上記のチェック項目に従い、各項目を検証。

### Step 5: デプロイ計画の作成

```markdown
# デプロイ計画

## Pass 1: 基本フィールド

| 順序 | アプリ名 | フィールド数 | 操作 |
|------|---------|-------------|------|
| 1 | 顧客マスタ | 10 | 作成→基本フィールド追加→デプロイ |
| 2 | 商品マスタ | 8 | 作成→基本フィールド追加→デプロイ |

## Pass 2: 連携フィールド（優先度順）

| 順序 | アプリ名 | フィールド名 | タイプ | 依存先 |
|------|---------|-------------|-------|--------|
| 1 | 受注管理 | 顧客名 | ルックアップ | 顧客マスタ.顧客ID |
| 2 | 受注管理 | 商品名 | ルックアップ | 商品マスタ.商品ID |

## 推定API呼び出し数

| 操作 | 回数 |
|------|------|
| アプリ作成 | X回 |
| フィールド追加 | Y回 |
| レイアウト更新 | Z回 |
| デプロイ | W回 |
| **合計** | **V回** |
```

以下の形式でレビュー結果を出力：

## 注意事項

1. **デプロイは非同期**: `deploy.json`を呼んだ後、ステータス確認が必要
2. **参照先は先にデプロイ**: ルックアップ/関連レコードの参照先が未デプロイだとエラー
3. **優先度順を厳守**: 参照先→参照元の順でフィールドを作成
4. **429エラー対策**: リトライ処理を実装しておく
5. **ロールバック不可**: kintoneは標準でロールバック機能がないため、慎重にデプロイ
6. **IPアドレス制限**: 外部からのAPI実行時は許可リストを確認
