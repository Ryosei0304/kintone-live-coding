# kintoneアプリ設計書生成

Phase 1で生成した概要書から、kintoneアプリ設計書を生成します。

## 概要

このスキルは `/kintone-workflow` から呼び出され、以下のファイルを生成します：
- `アプリ設計書_${Project}_${Date}.md` - アプリ一覧、アプリ間連携、要求機能一覧

## 入力要件

アプリ設計書を生成するには、以下の情報が必要です：

1. **概要書**（必須）: Phase 1で生成したシステム概要書・要件書・業務フロー
2. **プロジェクト固有の考え方**（任意）: プロジェクト特有のルールや制約

## 出力形式

マークダウン形式で以下の構造で出力します：

```markdown
# kintoneアプリ設計書

## アプリ一覧
| アプリ名 | 種別 | 説明 |
|----------|------|------|
| ${アプリ名} | ${マスタ/トランザクション} | ${アプリの説明} |

## アプリ間連携一覧
| 連携元アプリ | 連携先アプリ | 関係性 | キーフィールド | 参照フィールド |
|-------------|-------------|--------|----------------|----------------|
| ${連携元} | ${連携先} | ${ルックアップ/関連レコード一覧} | ${キー} | ${参照} |

## 要求機能一覧
| アプリ名 | 機能名 | 説明 | 実装方法 |
|--------|--------|------|----------|
| ${アプリ名} | ${機能名} | ${説明} | ${標準機能/プラグイン/カスタマイズ/外部サービス} |

## 考慮点
- ${考慮すべき事項}
```

## 情報不足・低確度時の対応

### 生成前チェック（必須）

生成を開始する前に、以下の情報が入力に含まれているか確認してください。不足・曖昧な項目がある場合は、**生成を開始せず、先にユーザーに質問してください**。

#### 必須確認項目

1. **アプリ分割方針**: どの単位でアプリを分けるか（業務単位、データ単位など）
2. **マスタ/トランザクション区分**: どのデータがマスタで、どのデータがトランザクションか
3. **権限モデル**: 誰がどのアプリ/レコード/フィールドにアクセスできるか
4. **プロセス管理の要否**: 承認フローやワークフローが必要な業務はどれか

#### 質問テンプレート（不足時に使用）

以下の質問を優先度順に行い、回答を得てから生成を開始してください：

1. 「アプリ分割の優先基準を教えてください（業務単位/データ単位/部門単位など）」
2. 「マスタデータとして管理するものを教えてください（顧客、商品、社員など）」
3. 「権限ロールを教えてください（管理者、一般ユーザー、閲覧のみなど）」
4. 「承認フローやワークフローが必要な業務はありますか？」

### 低確度時の対応

以下の場合は推測で埋めず、ユーザーに確認してください：

- 同じデータを複数アプリで持つべきか判断できない場合
- プロセス管理を採用すべきか判断できない場合
- アプリ間連携の方式（ルックアップ/関連レコード一覧）が不明な場合

**原則**: 不明な項目は「要確認」と記載し、仮の設計で埋めない。

## 生成手順

### 1. 概要書の分析

概要書から以下を抽出します：
- 必要なデータエンティティ
- 業務プロセス
- ユーザーの操作
- ユースケース

### 2. kintoneアプリ設計の原則

以下の原則に従ってアプリを設計します：

#### 禁止事項（アプリ化してはいけないもの）
- **帳票管理**: 各アプリに専用プラグインを設定するため不要
- **通知/リマインド専用**: 対象アプリにフィールドを含める
- **ユーザー管理**: kintone標準機能で対応
- **権限管理**: kintoneアクセス権で対応
- **分析/レポート/CSV専用**: kintone標準機能で対応
- **データ移行**: システム導入初期のみの作業のため不要

#### プロセス管理の判断基準
- **採用する場合**: 状態遷移・作業者・アクションが業務要件の本質（承認フロー、ワークフロー管理）
- **採用しない場合**: 通知・アクセス権・フィールド制御等で要件を満たせる場合

#### アプリ間連携フィールドの選択

| 連携タイプ | 用途 | 判断基準 |
|-----------|------|----------|
| ルックアップ | マスタからの情報転記・入力補助 | 入力時にマスタから1件を指定してコピーしたい場合 |
| 関連レコード一覧 | 関連情報の表示 | 閲覧時に別アプリの関連データを一覧表示したい場合 |
| アプリアクション | レコードの転送 | 登録済みレコードを他アプリに転送したい場合 |

### 3. 実装方法の判断

各機能について以下の順で実装方法を判断します：

1. kintone標準機能で実現可能 → **標準機能**
2. 既存プラグインで実現可能 → **プラグイン**
3. JavaScriptカスタマイズが必要 → **カスタマイズ**
4. 外部サービス連携が必要 → **外部サービス**

### 4. 考慮点の記載

以下の場合は「考慮点」セクションに記載します：
- 禁止事項に抵触する要求の代替方針
- 複雑なアクセス権が必要な場合の運用やカスタマイズの留意点
- プロセス管理を採用しない場合の理由（設計・保守・運用コストの観点で説明）
- 横断集計が必要な場合の設計上の妥当性と代替

## ファイル出力

生成したアプリ設計書は、以下の形式でファイルに保存してください：

- **ファイル名**: `アプリ設計書_${プロジェクト名}_${YYYYMMDD}.md`
- **保存先**: `outputs/${プロジェクト名}/`ディレクトリ
- **文字コード**: UTF-8

ファイル保存後、保存先のパスをユーザーに通知してください。

## セルフチェックリスト

生成後、以下を確認してください：

- [ ] 禁止事項に該当するアプリを作っていないか
- [ ] プロセス管理で済むものをアプリ化していないか
- [ ] 通知/帳票/分析/ユーザー/権限のためだけのアプリを作っていないか
- [ ] 連携は最小か（重複や循環・過度な分割がないか）
- [ ] 実装方法が適切に選択されているか
- [ ] 考慮点に必要な情報が記載されているか
- [ ] **全アプリにアプリの説明文（HTML）が記載されているか**
- [ ] **ファイルに保存したか**

## 影響範囲（下流ドキュメント）

アプリ設計書を更新・再生成した場合、以下のドキュメントの整合性確認・再生成が推奨されます：

```
アプリ設計書（このドキュメント）
  └→ フィールド設計書
```

## 使用しない条件

- 概要書がない場合（先にPhase 1を実行してください）
- kintone以外のプラットフォームでの設計
