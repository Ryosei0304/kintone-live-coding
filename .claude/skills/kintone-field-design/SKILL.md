# kintoneフィールド設計書生成

アプリ設計書からkintoneフィールド設計書を生成します。

## 概要

このスキルは `/kintone-workflow` から呼び出され、以下のファイルを生成します：
- `フィールド設計書_${Project}_${Date}.md` - 各アプリのフィールド定義、カスタマイズ設計

## 入力要件

フィールド設計書を生成するには、以下の情報が必要です：

1. **アプリ設計書**（必須）: アプリ一覧とアプリ間連携情報
2. **概要書**（推奨）: 実現したい機能の詳細

## 出力形式

マークダウン形式で以下の構造で出力します：

```markdown
# kintoneフィールド設計書

## アプリ一覧
| アプリ名 | 説明 |
|----------|------|
| ${アプリ名} | ${アプリの概要} |

## フィールド詳細
### ${アプリ名}

| フィールド名 | フィールドコード | フィールドタイプ | 必須 | 重複禁止 | 説明 | 備考 |
|--------------|------------------|------------------|------|----------|------|------|
| ${フィールド名} | ${フィールドコード} | ${フィールドタイプ} | ${○/-} | ${○/-} | ${説明} | ${備考} |

## カスタマイズ設計（該当する場合）
### ${アプリ名}

#### 適用パターン
| No | パターン | 設定 | 要件 |
|----|---------|------|------|
| 1 | ${パターンID} | ${設定内容} | ${要件の説明} |

#### スペーサー定義（style_section_header使用時）
| スペーサーID | 配置位置 | 用途 |
|-------------|---------|------|
| ${space_xxx} | ${配置位置} | ${用途} |
```

## 情報不足・低確度時の対応

### 生成前チェック（必須）

生成を開始する前に、以下の情報が入力に含まれているか確認してください。不足・曖昧な項目がある場合は、**生成を開始せず、先にユーザーに質問してください**。

#### 必須確認項目

1. **フィールドの型**: 各フィールドのデータ型（文字列、数値、日付など）
2. **必須/任意**: どのフィールドが必須入力か
3. **選択肢**: ドロップダウン、ラジオボタン、チェックボックスの選択肢一覧
4. **ルックアップ設定**: 参照元アプリ、キーフィールド、コピーフィールド

#### 質問テンプレート（不足時に使用）

以下の質問を優先度順に行い、回答を得てから生成を開始してください：

1. 「選択肢フィールドの選択肢一覧を教えてください」
2. 「必須入力のフィールドはどれですか？」
3. 「入力桁数や形式の制限があるフィールドはありますか？」
4. 「ルックアップの参照元アプリとキーフィールドを教えてください」

### 低確度時の対応

以下の場合は推測で埋めず、ユーザーに確認してください：

- 選択肢が決まっていない場合
- フィールド名と業務用語の対応が不明な場合
- ルックアップのキーフィールドが不明な場合
- ユニーク制約が必要かどうか不明な場合

**原則**: 選択肢や制約が未確定の場合は「要確認」と記載し、仮の値で埋めない。

## 生成手順

### 1. アプリ設計書の分析

アプリ設計書から以下を抽出します：
- アプリ一覧
- アプリ間連携情報（ルックアップ、関連レコード一覧）
- 要求機能一覧

### 2. フィールド設計の原則

#### フィールドコードの命名規則
- フィールドコードは英数字とアンダースコアのみ使用（スネークケース）
- フィールド名（日本語ラベル）とは別に、プログラム用の英語コードを設定する
- 命名パターン:
  - 基本項目: customer_name, order_date
  - 金額系: total_amount, unit_price
  - フラグ系: is_approved, is_active
  - ルックアップ: customer_lookup, product_lookup
  - 計算: calc_tax, calc_total
  - テーブル内: detail_product_name, detail_quantity

#### フィールドタイプの選択
- フィールドタイプは必ず以下のkintoneフィールドタイプから選択すること
- データベース用語（VARCHAR、INT、DATETIMEなど）は使用しない

#### 数値フィールドの備考について
- 金額（円）、個数、数量など整数で管理するフィールドには「小数点」に関する備考は記載しない
- 小数点の指定が必要なのは、外貨金額、割合（パーセント）、重量（kg）など明確に小数が必要な場合のみ
- 日本円の金額フィールドは整数として扱うため、小数点の備考は不要

### 3. kintoneフィールドタイプ一覧

#### テキストフィールド
- **文字列（1行）**: 単一行のテキスト入力
- **文字列（複数行）**: 複数行のテキスト入力
- **リッチエディター**: フォーマット付きテキスト入力

#### 数値フィールド
- **数値**: 数値データの入力

#### 選択肢フィールド
- **ラジオボタン**: 複数の選択肢から1つを選ぶ（選択肢が少ない場合）
- **チェックボックス**: 複数の選択肢から複数を選ぶ（選択肢が少ない場合）
- **複数選択**: 複数の選択肢から複数を選ぶ（選択肢が多い場合）
- **ドロップダウン**: ドロップダウンメニューから1つを選ぶ（選択肢が多い場合）
- **ユーザー選択**: ユーザーを選択
- **組織選択**: 組織を選択
- **グループ選択**: グループを選択

#### 日時フィールド
- **日付**: 日付データの入力
- **時刻**: 時刻データの入力
- **日時**: 日付と時刻データの入力

#### その他
- **リンク**: URLやメールアドレスなどのリンクデータ
- **添付ファイル**: ファイルを添付
- **テーブル**: 複数行データを表形式で管理

#### アプリ間連携フィールド
- **ルックアップ**: 他のアプリからデータを参照
- **関連レコード一覧**: 関連するレコードを一覧表示

### 4. 除外すべきシステムフィールド

以下のシステムフィールドは自動生成されるため、設計書に含めてはいけません：
- ステータス（STATUS）
- 作業者（STATUS_ASSIGNEE）
- カテゴリー（CATEGORY）
- 作成者（CREATOR）
- 作成日時（CREATED_TIME）
- 更新者（MODIFIER）
- 更新日時（UPDATED_TIME）
- レコード番号（RECORD_NUMBER）

#### プロセス管理利用時の注意

アプリアーキテクチャでプロセス管理が指定されているアプリでは:
- ステータス用のカスタムフィールド（ドロップダウン等）を**作成してはいけない**
- kintoneのプロセス管理のSTATUSシステムフィールドを使用する
- アーキテクチャ書でステータスが「プロセス管理（システムフィールド）」と記載されている場合、フィールド一覧に含めない

### 5. アプリ間連携フィールドの注意点

- ルックアップフィールドの「コピー元のフィールド」や「ほかのフィールドのコピー」には、テーブル内のフィールドは指定できません
- 関連レコード一覧フィールドの「表示するレコードの条件」には、テーブル内のフィールドは指定できません
- アプリ間連携フィールドは配置されるアプリにのみ設けるべきで、ルックアップ・関連レコード先のアプリに同様のフィールドは設けないこと

### 6. ルックアップ制約（重要）

ルックアップのキーフィールド（参照先アプリのフィールド）には**必ず `重複禁止: ○`（unique: true）を設定**すること。
この設定がないと、データ投入時に `GAIA_LO03` エラーが発生する。

```
例: 顧客マスタ → 企業マスタ（会社名でルックアップ）
→ 企業マスタの「会社名」フィールドに 重複禁止: ○ を設定
```

### 7. ラベル見出し・入力ガイドラベルの自動提案

#### スペーサー+セクション見出し
- フィールドが5つ以上のアプリでは、論理グループごとにスペーサー+セクション見出しを設計すること
- スペーサーID命名規則: `space_{セクション名}` (例: `space_basic_info`)
- `style_section_header` カスタマイズパターンとセットで運用

#### 入力ガイドラベル
以下の条件に該当するフィールドには、入力ガイドラベルを自動提案すること：
- ルックアップフィールドの上 → 「取得ボタンを押して選択してください」
- 必須の日付フィールド → 「YYYY/MM/DD形式で入力」
- 電話番号リンクフィールド → 「ハイフン付きで入力（例: 03-1234-5678）」
- 金額フィールド（税込計算がある場合） → 「税抜金額を入力してください」

### 8. ビュー自動提案ルール

フィールド設計に基づき、以下のルールでビューを自動提案すること：

- ステータス系フィールド（ドロップダウン/ラジオボタンで状態を表すもの）がある場合 → 各ステータス値ごとのフィルタビューを提案
- 担当者フィールド（ユーザー選択）がある場合 → 「自分の担当」ビュー（`担当者 in (LOGINUSER())`）を提案
- 日付フィールドがある場合 → 「今月分」ビュー（`日付 >= THIS_MONTH()`）を提案

### 9. カスタマイズ設計

概要書のカスタマイズ要件から、JavaScript/CSSによるカスタマイズを設計。

**対応パターン**:

| パターンID | 名前 | 用途 |
|-----------|------|------|
| `field_show_hide` | フィールド表示/非表示 | 条件に応じてフィールドを表示/非表示 |
| `field_disable` | フィールド編集不可 | 条件に応じてフィールドを編集不可 |
| `condition_status` | ステータス分岐 | ステータス値に応じた複合処理 |
| `style_section_header` | セクション見出し | スペーサーに見出しを追加 |
| `validate_required` | 条件付き必須 | 条件に応じて必須チェック |

**要件マッチングの例**:

```
要件: 「ステータスが完了の時、備考フィールドを編集不可にしたい」
↓
パターン: field_disable
設定:
  - TRIGGER_FIELD: "ステータス"
  - CONDITION_VALUE: "完了"
  - TARGET_FIELDS: ["備考"]
  - DISABLE_WHEN_MATCH: true
```

**未対応要件の記録**:

パターンで対応できない要件は明記する：
```
### 未対応カスタマイズ要件
| 要件 | 理由 | 代替案 |
|------|------|--------|
| 金額が10万円以上の時、承認者を表示 | 数値比較パターン未対応 | カスタム開発が必要 |
```

## ファイル出力

生成したフィールド設計書は、以下の形式でファイルに保存してください：

- **ファイル名**: `フィールド設計書_${プロジェクト名}_${YYYYMMDD}.md`
- **保存先**: `outputs/${プロジェクト名}/`ディレクトリ
- **文字コード**: UTF-8

ファイル保存後、保存先のパスをユーザーに通知してください。

## セルフチェックリスト

生成後、以下を確認してください：

- [ ] フィールドコードがスネークケース命名規則に従っているか
- [ ] フィールドタイプがkintoneの正式名称で記載されているか
- [ ] システムフィールドが含まれていないか
- [ ] アプリ間連携フィールドが適切に設計されているか
- [ ] テーブル内フィールドにルックアップ・関連レコード一覧を設定していないか
- [ ] ルックアップのキーフィールドに `重複禁止: ○` が設定されているか
- [ ] 日本円の金額フィールドに不要な小数点の備考がないか
- [ ] カスタマイズ設計が適切にパターンマッピングされているか
- [ ] **ファイルに保存したか**

## テストデータ投入時の注意点

### APIで登録できないフィールド設定

| フィールド種別 | 問題 | 対策 |
|---------------|------|------|
| USER_SELECT（必須） | `LOGINUSER()`デフォルト値がAPI経由で効かない | 必須設定を避けるか、ユーザーコードを明示指定 |
| LOOKUP（必須） | 参照先のキーフィールドにunique設定が必須 | デプロイ時にunique設定を確認 |
| STATUS/STATUS_ASSIGNEE | プロセス管理はAPIで設定不可 | 手動設定または別途REST API |

### 推奨: 必須設定を避けるべきフィールド

テストデータ投入フェーズでのエラーを防ぐため、以下は**必須にしない**ことを推奨：

- `USER_SELECT`（担当者、承認者など）
- `LOOKUP`（ルックアップフィールド）← 参照先のunique設定が前提
- `ORGANIZATION_SELECT`
- `GROUP_SELECT`

## 影響範囲（下流ドキュメント）

フィールド設計書を更新・再生成した場合、以下のステップに影響します：

```
フィールド設計書（このドキュメント）
  └→ Phase 3: kintone app creation
```

## 使用しない条件

- アプリ設計書がない場合（先にkintone-app-designを実行してください）
- kintone以外のプラットフォームでの設計
