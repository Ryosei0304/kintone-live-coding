# Phase 2: 業務フロー整理（WHAT / WHEN）

コンテキスト整理書をもとに、業務フローの現状と改善後を設計し、アプリ境界の基礎となるビジネスイベントを洗い出します。

## 概要

このスキルは `/kintone-workflow` から呼び出され、以下を実行します：
1. コンテキスト整理書を読み込む
2. As-Is業務フローを推測して提案 → ユーザーに修正してもらう
3. To-Beフロー（kintone導入後）をMermaidで提案
4. ビジネスイベントを洗い出す
5. MVPスコープ線引き
6. 規模感・連携・セキュリティの確認
7. カスタマイズ提案（推奨度付き）

出力: `業務フロー設計_${Project}_${Date}.md`

## 入力

- `${OutputDir}コンテキスト整理_${Project}_${Date}.md`（最初に読み込むこと）
- プロジェクト名
- 出力日付（YYYYMMDD形式）

## 分析方針（重要）

**「聞く」より「見せて直す」方が速い。**

コンテキストから業務フローを**推測して提案**し、ユーザーに修正してもらう方式。

詳細は `.claude/agents/reference/flow-analysis-guide.md` を参照。

## 実行手順

### Step 1: コンテキスト読み込み

`コンテキスト整理_${Project}_${Date}.md` を読み込み、アクター・ペインポイント・ゴール・スコープを把握。

### Step 2: As-Is業務フロー推測 → 提案

コンテキストから現状の業務フローを推測し、Mermaid sequenceDiagram で提案:

```
お話を聞いた内容から、今の業務の流れをこんな感じで理解しました！
[Mermaid図]
合っていますか？違うところがあれば教えてください！
```

→ ユーザーの修正を反映してAs-Isフローを確定

### Step 3: To-Beフロー提案

kintone導入後のTo-Beフローを提案:

```
kintoneを使うと、こんな流れになります！
[Mermaid図]
改善ポイント:
1. ...
2. ...
この流れでよさそうですか？
```

### Step 4: ビジネスイベント洗い出し

To-Beフローから「データが生まれるトリガー」を洗い出す:

```
この業務で「データが生まれるタイミング」を整理しました。
これらがアプリの元になります！
| No | イベント名 | トリガー | 生成されるデータ |
漏れがないか確認してもらえますか？
```

### Step 5: MVPスコープ線引き

| 判定 | 条件 |
|------|------|
| MVP | ペインポイントの直接解決に必要 |
| MVP | 業務が回るための最低限の機能 |
| 将来 | あると便利だが、なくても業務は回る |

### Step 6: 規模感・連携・セキュリティ確認

業務フローが固まった段階で初めて聞く:

```
業務フローが整理できたので、あと少しだけ確認させてください！
1. 利用者数はだいたい何人くらいですか？
2. データは月にどのくらい増えそうですか？
3. 他のシステムとの連携は必要ですか？
4. アクセス制限で気をつけることはありますか？
分からないものは「分からない」で大丈夫です！
```

### Step 7: カスタマイズ提案

業務特性から適用可能なカスタマイズを推奨度付きで提案。
パターンカタログ（`templates/customize/catalog.json`）を参照。

**推奨度判定ロジック**: `reference/flow-analysis-guide.md` の「カスタマイズ提案」セクションを参照。

### Step 8: 業務フロー設計書生成

テンプレート `templates/flow-design-template.md` に従い生成:

- `${OutputDir}業務フロー設計_${Project}_${Date}.md`

## セルフチェックリスト

- [ ] As-Is/To-Beの両方がMermaidフロー図付きで記述されている
- [ ] ビジネスイベントが漏れなく洗い出されている
- [ ] MVPスコープが明確に線引きされている
- [ ] 規模感・連携・セキュリティが記載されている
- [ ] カスタマイズ提案が推奨度付きで記載されている
- [ ] ユーザーのカスタマイズ選択結果が記録されている

## 注意事項

1. **推測して見せる**: 漠然と質問するのではなく、推測結果を見せて修正してもらう
2. **ビジネスイベント = アプリの元**: 「データが生まれるタイミング」がPhase 3でのアプリ境界決定の基礎になることを意識
3. **カスタマイズは業務理解が新鮮なうちに**: 業務フロー整理直後が最適なタイミング
4. **規模感は概算で十分**: 正確な数値より桁感が分かればOK
