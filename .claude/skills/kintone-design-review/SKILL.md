# Phase 2: 設計書生成前インプットレビュー

Phase 1の概要書とHITLで収集した追加情報が、Phase 2（設計書生成）に十分かをkintoneエキスパートの観点でレビューします。

## 概要

このスキルはPhase 2（設計書生成）の**前**に実行され、以下を検証します：

- 概要書からアプリ分割に必要な情報が揃っているか
- 連携要件（ルックアップ・関連レコード）が明確か
- ワークフロー要件の明確性
- カスタマイズ要件の明確性

## 入力

- Phase 1で生成された概要書ファイル
  - `システム概要書_${Project}_${Date}.md`
  - `機能要件書_${Project}_${Date}.md`
  - `業務フロー_${Project}_${Date}.md`
- HITL #1 で収集した追加情報（コンテキスト上の会話）

## レビュー観点

### 1. アプリ分割に必要な情報の充足度

| チェック項目 | 観点 | 重要度 |
|-------------|------|--------|
| マスタとトランザクションの区別が可能か | 概要書からデータの性質（マスタ/トランザクション）が判断できるか | 必須 |
| アプリの粒度が判断できるか | 1アプリ1目的に分割するための情報が十分か | 必須 |
| アプリ間の関係が推測できるか | どのデータがどのデータを参照するか判断できるか | 必須 |
| 再利用性の判断が可能か | マスタデータを複数アプリから参照する必要性が把握できるか | 推奨 |

### 2. 連携要件の明確性

| チェック項目 | 観点 | 重要度 |
|-------------|------|--------|
| 参照関係が特定できるか | どのデータがどのデータを参照するか明確か | 必須 |
| ルックアップ vs 関連レコードの判断材料があるか | コピー保持 or リアルタイム参照の要件が分かるか | 必須 |
| キーフィールドの候補が推測できるか | ルックアップのキーとなるフィールドが特定できるか | 推奨 |

### 3. プロセス管理（ワークフロー）要件の明確性

ワークフロー要件がある場合に検証：

| チェック項目 | 観点 | 重要度 |
|-------------|------|--------|
| ステータス遷移が把握できるか | 業務フローからステータス一覧が導出できるか | 必須 |
| 各ステータスの作業者が推測できるか | 誰がどのステータスで作業するか分かるか | 必須 |
| アクションが特定できるか | 承認・差戻し等のアクションが明確か | 推奨 |

### 4. カスタマイズ要件の明確性

カスタマイズ要件がある場合に検証：

| チェック項目 | 観点 | 重要度 |
|-------------|------|--------|
| パターンが特定できるか | 表示/非表示、編集不可、必須チェック等のパターンが特定可能か | 必須 |
| 対象フィールドが推測できるか | カスタマイズ対象のフィールドが概要から特定できるか | 必須 |
| 条件が明確か | トリガー条件（ステータス、フィールド値等）が具体的か | 必須 |
| kintone標準では不可能か | 標準機能で代替できないことが確認できるか | 推奨 |

## レビュー実行手順

### Step 1: ファイル読み込み

outputs/ディレクトリから概要書ファイルを読み込み、コンテキスト上の追加情報も参照する。

### Step 2: チェック項目の検証

上記のチェック項目に従い、各項目の充足度を検証。

### Step 3: 不足情報の特定

以下のカテゴリで不足情報を分類：

| カテゴリ | 説明 |
|---------|------|
| CRITICAL | この情報がないと設計書を生成できない |
| WARNING | なくても生成可能だが、設計品質が低下する |
| INFO | あるとより良い設計書になる |

### Step 4: レビュー結果の出力

以下の形式でレビュー結果を出力：

```markdown
# Phase 2 インプットレビュー結果

## サマリー

| カテゴリ | 件数 |
|---------|------|
| CRITICAL | X件 |
| WARNING | Y件 |
| INFO | Z件 |

## インプット充足度

| チェック項目 | 状態 | 備考 |
|-------------|------|------|
| マスタ/トランザクション区別 | OK/不足 | ... |
| アプリ粒度の判断材料 | OK/不足 | ... |
| アプリ間関係 | OK/不足 | ... |
| 連携要件 | OK/不足 | ... |
| ワークフロー要件 | OK/不足/該当なし | ... |
| カスタマイズ要件 | OK/不足/該当なし | ... |

## 詳細

### CRITICAL（不足情報）

#### [C-001] {不足情報のタイトル}
- **不足内容**: {何が足りないか}
- **なぜ必要か**: {この情報がないとどうなるか}
- **追加質問**: {ユーザーに聞くべき質問}

### WARNING

#### [W-001] {不足情報のタイトル}
...

### INFO

#### [I-001] {改善提案のタイトル}
...

## 次のステップ

- CRITICALがある場合: 追加質問をユーザーに提示し、回答を収集
- CRITICALがない場合: Phase 2（設計書生成）に進む
```

## 注意事項

1. **生成前のチェックである**: フィールド設計・デプロイ順序など、まだ生成されていない情報はチェック対象外
2. **概要レベルで十分**: 詳細なフィールド名やコードは設計時に決まるため、概要レベルの情報があればOK
3. **ユーザー負担の最小化**: CRITICALの追加質問は具体的かつ最小限に絞る
4. **既存概要書を活用**: 概要書に記載済みの情報は重複して質問しない
