# Phase 1: 収集情報の充足度レビュー

ユーザーから収集した情報がPhase 1（概要書生成）に十分かをkintoneエキスパートの観点でレビューします。

## 概要

このスキルはPhase 1（概要書生成）の**前**に実行され、以下を検証します：

- 収集情報から業務フローが組み立て可能か
- kintoneへの適合性
- 概要書生成に必要な情報が揃っているか

## 入力

- コンテキスト上の収集済みユーザー入力情報（会話の中で収集した業務要件・データ項目・利用者情報・カスタマイズ要件など）

## レビュー観点

### 1. 業務フロー組み立て可能性

| チェック項目 | 観点 | 重要度 |
|-------------|------|--------|
| 業務の始まりと終わりが推測できるか | 収集情報から業務の開始点・終了点が特定できるか | 必須 |
| 登場人物（アクター）が特定できるか | 誰が、どの部署で、どんな操作をするか推測できるか | 必須 |
| 「ヒト」「モノ」「データ」の流れが把握できるか | 収集情報から業務フローの骨格が組み立て可能か | 必須 |
| 主要フローをカバーする情報があるか | 8割の状況をカバーできる程度の情報が揃っているか | 推奨 |

### 2. kintone適合性判定

#### kintoneに向いている業務

| 業務タイプ | 適合理由 |
|-----------|---------|
| 顧客管理・案件管理・タスク管理 | 標準機能でほぼ実現可能 |
| ワークフロー・申請承認 | プロセス管理機能で対応 |
| 日報・報告書管理 | フォーム・コメント機能で対応 |
| マスタデータ管理 | ルックアップ・関連レコードで連携可能 |

#### kintoneに向いていない/要検討の業務

| 業務タイプ | 理由 | 代替案 |
|-----------|------|--------|
| 高度なデータ分析 | 複雑な集計・数千件以上の高速処理が苦手 | BIツール連携を検討 |
| 会計処理・税務計算 | 専門性・制度的要件が必要 | 専用システムとの連携 |
| リアルタイムトランザクション処理 | 予約・決済など原子性が必要な処理は不向き | RDBMSを検討 |
| 複雑な権限管理 | 行レベルセキュリティは標準では限定的 | カスタマイズまたは別システム |

### 3. 収集情報の完成度チェック

#### 必須情報

- [ ] システムの目的・背景が把握できる情報がある
- [ ] 対象業務の範囲（何をやり、何をやらないか）が推測できる
- [ ] 利用者の種類と役割が特定できる
- [ ] 必要なデータ項目の概要が把握できる

#### 推奨情報

- [ ] 想定利用者数に関する情報がある
- [ ] 想定データ量に関する情報がある
- [ ] データの関連性（1:1、1:N、N:M）が推測できる
- [ ] 外部連携の要否に関する情報がある
- [ ] セキュリティ要件に関する情報がある

## レビュー実行手順

### Step 1: 収集情報の整理

会話コンテキストからユーザーが提供した情報を整理する。

### Step 2: チェック項目の検証

上記のチェック項目に従い、各項目の充足度を検証。

### Step 3: 不足情報の特定

以下のカテゴリで不足情報を分類：

| カテゴリ | 説明 |
|---------|------|
| CRITICAL | この情報がないと概要書を生成できない |
| WARNING | なくても生成可能だが、品質が低下する |
| INFO | あるとより良い概要書になる |

### Step 4: レビュー結果の出力

以下の形式でレビュー結果を出力：

```markdown
# Phase 1 収集情報レビュー結果

## サマリー

| カテゴリ | 件数 |
|---------|------|
| CRITICAL | X件 |
| WARNING | Y件 |
| INFO | Z件 |

## kintone適合性判定

**判定結果**: 適合 / 要検討 / 不適合

**理由**:
- ...

## 収集情報の充足度

| チェック項目 | 状態 | 備考 |
|-------------|------|------|
| 目的・背景 | OK/不足 | ... |
| 業務範囲 | OK/不足 | ... |
| 利用者・役割 | OK/不足 | ... |
| データ項目 | OK/不足 | ... |

## 詳細

### CRITICAL（不足情報）

#### [C-001] {不足情報のタイトル}
- **不足内容**: {何が足りないか}
- **なぜ必要か**: {この情報がないとどうなるか}
- **追加質問**: {ユーザーに聞くべき質問}

### WARNING

#### [W-001] {不足情報のタイトル}
...

### INFO

#### [I-001] {改善提案のタイトル}
...

## 次のステップ

- CRITICALがある場合: 追加質問をユーザーに提示し、回答を収集
- CRITICALがない場合: Phase 1（概要書生成）に進む
```

## 注意事項

1. **完璧を求めすぎない**: 8割の状況をカバーできる情報があれば十分
2. **kintoneの限界を認識**: 不向きな業務は早期に判断し、代替案を提示
3. **ユーザー負担の最小化**: CRITICALの追加質問は具体的かつ最小限に絞る
4. **トレーサビリティ**: 不足と判定した理由を明確に記載する
