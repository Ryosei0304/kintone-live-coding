# Phase 1: システム概要書生成

収集したインプット情報から、kintoneシステムの概要書を生成します。

## 概要

このスキルは `/kintone-workflow` から呼び出され、以下の3ファイルを生成します：
- `システム概要書_${Project}_${Date}.md` - 背景 + 目的
- `機能要件書_${Project}_${Date}.md` - 機能要件 + 非機能要件
- `業務フロー_${Project}_${Date}.md` - 業務フロー

## 入力

- 収集済みのユーザー入力情報
- プロジェクト名
- 出力日付（YYYYMMDD形式）

## 生成前チェックリスト

生成を開始する前に、以下の情報が揃っていることを確認：

### ヒアリング方針（重要）

**質問は一度に全て聞かず、セクションごとに2〜3問ずつ分割して聞くこと。**
ユーザーの認知負荷を減らし、回答しやすくする。

#### ヒアリングの流れ

1. **Section 1: 基本情報**（2問）→ 回答を待つ
2. **Section 2: 利用者情報**（2問）→ 回答を待つ
3. **Section 3: 規模感**（2問）→ 回答を待つ
4. **Section 4: 連携・セキュリティ**（2問）→ 回答を待つ
5. **Section 5: カスタマイズ提案**（提案付きヒアリング）→ 回答を待つ

各セクションの回答後に簡潔にサマリーして確認し、不明点があればその場で深掘りする。

#### Section 5 カスタマイズ提案（重要）

Section 1〜4で収集した業務内容から、適用可能なカスタマイズを推奨度付きで提案する。
「何かカスタマイズありますか？」と漠然と聞くのではなく、**具体的な提案を見せて選んでもらう**。

**推奨度の3段階:**

| 推奨度 | 意味 | 判定基準 |
|--------|------|----------|
| ★★★ 強く推奨 | 運用上ほぼ必須。ないと困る場面が多い | ステータス管理あり→完了時の編集ロック、承認フローあり→条件付き必須 |
| ★★ 推奨 | この業務特性に合っている | 入力項目が多い→セクション見出し、連絡先管理→フォーマットチェック |
| ★ あると便利 | なくても回るが、あると嬉しい | 一覧の色分け、文字数カウント、自動採番 |

**提案ロジック（Section 1〜4の回答から判定）:**

| 業務特性のヒント | 提案するカスタマイズ | 推奨度 |
|-----------------|-------------------|--------|
| ステータス管理・承認フローがある | 完了時にフィールド編集不可 | ★★★ |
| ステータスに応じて入力項目が変わる | ステータスで表示/非表示を切替 | ★★★ |
| 条件によって必須項目が変わる | 条件付き必須チェック | ★★★ |
| 管理項目が多い（10項目以上）業務 | セクション見出しで区切り | ★★ |
| 電話番号・メール・郵便番号を扱う | 入力フォーマットチェック | ★★ |
| 明細行（見積明細、注文明細等）がある | テーブル合計計算 | ★★ |
| 採番ルールがある（例: INV-001） | 自動採番 | ★★ |
| 入社日・契約開始日など期間管理 | 経過年数/日数表示 | ★ |
| 一覧画面でステータスを素早く把握したい | 一覧画面の条件書式（色分け） | ★ |
| 備考欄など自由入力が多い | 文字数カウント表示 | ★ |

**表示フォーマット:**

```
お話を聞いた内容から、こんなカスタマイズが効果的だと思います！

| # | 推奨度 | カスタマイズ | 内容 | 理由 |
|---|--------|------------|------|------|
| 1 | ★★★ | フィールド編集不可 | ステータスが完了のとき全フィールドを編集不可に | 承認済みデータの誤編集を防止できます |
| 2 | ★★ | セクション見出し | 「基本情報」「連絡先」「対応履歴」で区切り | 管理項目が多いので、セクションで区切ると見やすくなります |
| 3 | ★ | 一覧の条件書式 | ステータスごとに行の色を変更 | 対応状況がひと目で分かるようになります |

この中で「いいね！」と思うものはありますか？
他にも「こういうことできる？」があれば気軽に聞いてください！
```

**回答の処理:**
- 選択されたカスタマイズ → 概要書のカスタマイズ要件セクションに記載
- 追加要望 → パターンカタログ（`templates/customize/catalog.json`）と照合し、対応可否を回答
- 全て不要 → カスタマイズ要件なしとして記録

### 必須確認項目

| カテゴリ | セクション | 確認項目 | 質問テンプレート | 重要度 |
|----------|------------|----------|------------------|--------|
| スコープ | Section 1 | 対象業務範囲（何をやり、何をやらないか） | 「どんな業務をシステム化したいですか？今困っていることがあれば教えてください」 | 必須 |
| データ | Section 1 | 管理したいデータ・情報 | 「このシステムで管理したいデータや情報は何ですか？」 | 必須 |
| 利用者 | Section 2 | ユーザー種別 | 「このシステムを使う人は誰ですか？（部署・役職など）」 | 必須 |
| 権限 | Section 2 | 権限レベル | 「ユーザーによって見せたい情報や操作権限に違いはありますか？」 | 必須 |
| 規模 | Section 3 | 利用者数 | 「想定している利用者数を教えてください」 | 推奨 |
| 規模 | Section 3 | データ件数 | 「データは1日/月あたりどのくらい増えそうですか？」 | 推奨 |
| 外部連携 | Section 4 | 連携システムの有無と内容 | 「他のシステムと連携する必要はありますか？」 | 推奨 |
| セキュリティ | Section 4 | アクセス制御・データ保護要件 | 「アクセス制限や個人情報の取り扱いで気をつけることはありますか？」 | 推奨 |
| カスタマイズ | Section 5 | 動作・見た目のカスタマイズ要件 | ※下記「Section 5 カスタマイズ提案」参照 | 推奨 |

### 情報充足度の判断基準

#### 十分と判断する条件
- スコープと利用者が明確に定義されている
- 主要な業務フローが説明されている
- 管理したいデータ項目が概ね把握できる

#### 追加質問が必要な条件
- 上記必須項目のいずれかが不明確
- 業務フローに曖昧な点がある
- ユーザーの役割分担が不明確

### 情報不足時の対応

1. 不足している情報を特定
2. 質問テンプレートを参考に、具体的な質問を生成
3. ユーザーからの回答を待つ
4. 回答を既存情報と統合
5. 再度チェックを実施

### サマリ表示フォーマット

情報が十分と判断した場合、以下の形式でサマリを表示：

```
## 収集情報サマリ

### システム概要
- 目的: [一言で]
- 対象業務: [列挙]
- 対象外: [列挙]

### 利用者
- [ユーザー種別1]: [役割と権限]
- [ユーザー種別2]: [役割と権限]

### 規模感

- 想定データ量: [件数/期間]

### カスタマイズ要件
- [動作制御の要件があれば記載]
- [見た目装飾の要件があれば記載]

### 特記事項
- [あれば記載]
```

## 生成手順

### 1. テンプレート読み込み

`templates/proposal-template.md` を参照し、構造を理解する。

### 2. 情報の整理・構造化

収集した情報を以下のカテゴリに分類：

**背景・目的**:
- 現状の課題
- システム化の背景
- 期待される効果
- スコープ（対象/対象外）

**機能要件**:
- ユースケース一覧
- 要求機能一覧
- 各機能の実装方法（kintone標準/プラグイン/カスタマイズ/外部連携）

**非機能要件**:
- 利用規模（人数、データ量）
- セキュリティ要件
- 運用要件
- 外部連携

**業務フロー**:
- 全体の流れ
- 各業務の詳細
- ステータス定義
- 承認フロー（該当する場合）

**カスタマイズ要件**:
- 動作制御（表示/非表示、編集可否）
- バリデーション（条件付き必須チェック）
- 見た目装飾（セクション見出し、強調表示）
- 自動処理（値の自動セット、自動計算）

### 3. ユースケーステーブルの作成

以下の形式でユースケースを整理：

| ID | ユースケース名 | アクター | 概要 | 優先度 |
|----|----------------|----------|------|--------|
| UC-001 | ... | ... | ... | 高/中/低 |

**ユースケース詳細の記述項目**:
- アクター
- 事前条件
- 基本フロー（ステップ形式）
- 事後条件
- 代替フロー
- 例外フロー

### 4. 実装方法の判断

各機能について、以下の基準で実装方法を判断：

| 実装方法 | 判断基準 |
|----------|----------|
| kintone標準 | kintoneの標準機能で実現可能 |
| プラグイン | 既存プラグインで実現可能（要調査） |
| カスタマイズ | JavaScript等の開発が必要 |
| 外部サービス | kintone外のサービス連携が必要 |

### 5. ファイル生成

Writeツールを使用して3ファイルを生成。

**ファイル1: システム概要書_{Project}_{Date}.md**
```markdown
# {ProjectName} システム概要書

作成日: {Date}

## 1. 背景
### 1.1 現状の課題
### 1.2 システム化の背景

## 2. 目的
### 2.1 システム導入の目的
### 2.2 期待される効果
### 2.3 スコープ

## 3. 利用者
```

**ファイル2: 機能要件書_{Project}_{Date}.md**
```markdown
# {ProjectName} 機能要件書

作成日: {Date}

## 1. 機能要件
### 1.1 ユースケース一覧
### 1.2 ユースケース詳細
### 1.3 要求機能一覧

## 2. 非機能要件
### 2.1 利用規模
### 2.2 セキュリティ要件
### 2.3 運用要件
### 2.4 外部連携
```

**ファイル3: 業務フロー_{Project}_{Date}.md**
```markdown
# {ProjectName} 業務フロー

作成日: {Date}

## 1. 業務フロー概要
### 1.1 全体像
### 1.2 業務フロー図

## 2. 業務詳細

## 3. ステータス定義

## 4. 承認フロー（該当する場合）
```

### 6. 生成結果の表示

生成完了後、以下を表示：
- 生成したファイル一覧
- 各ファイルの概要（セクション数、ユースケース数など）

## セルフチェックリスト

生成した概要書の品質を確認：

### 背景・目的
- [ ] 課題が具体的に記述されている
- [ ] 目的が明確に定義されている
- [ ] スコープ（対象/対象外）が明記されている

### 機能要件
- [ ] 全てのユースケースがIDで管理されている
- [ ] 各ユースケースのアクターが明確
- [ ] 基本フローがステップ形式で記述されている
- [ ] 実装方法が判断されている

### 非機能要件
- [ ] 利用規模（人数、データ量）が記載されている
- [ ] セキュリティ要件が検討されている

### 業務フロー
- [ ] 全体の流れが把握できる
- [ ] ステータス遷移が定義されている（該当する場合）

## 技術用語の説明

概要書内で使用する際は括弧書きで補足：

- **ユースケース**: システムの利用シーン。誰が何をするかを定義したもの
- **アクター**: システムを操作する人や外部システム
- **kintone標準機能**: kintoneに最初から備わっている機能
- **プラグイン**: kintoneの機能を拡張するアドオン
- **カスタマイズ**: JavaScriptなどで独自に開発する機能

## 注意事項

1. **曖昧な表現の回避**: 「多くの」「適切な」などは具体的な数値や条件に置き換える
2. **技術用語の補足**: 非エンジニア向けに括弧書きで説明を追加
3. **一貫性**: 用語や表記を統一する
4. **トレーサビリティ**: ユースケースIDと機能IDを紐付ける
